# 今日功能复盘（2026-03-20）

## 1. 今日学习主线
今天不是“堆新页面”，而是把已有项目做成更像真实业务项目的结构。核心路线是：
1. 把商品领域逻辑继续从页面抽离到 composable。
2. 把本地 mock 切换到真实第三方接口（Fake Store API）。
3. 打通登录鉴权链路（登录、守卫、请求拦截、401处理）。
4. 升级列表交互体验（排序、分页、首次加载与刷新状态分离）。

这条路线的目标是：页面只管展示和交互，数据获取与业务逻辑下沉到可复用层。

## 2. 第二轮抽离：商品逻辑进入 useProducts
### 做了什么
1. 在 `useProducts` 中统一管理商品列表状态。
2. 页面筛选（搜索、分类）逻辑沉到 composable。
3. 商品统计逻辑（电子产品数量、平均价格等）集中管理。
4. 商品查询能力（按 id 查找）从页面抽离为统一函数。

### 为什么这样做
1. 避免 `PractiseView` 和 `PractiseDetailsView` 重复写同类逻辑。
2. 页面更薄，后续接 API 或改规则时只改一处。
3. 形成“领域逻辑可复用”的结构，后续其他页面也能直接用。

### 关键认知
1. 页面层只消费状态和方法，不直接持有业务规则。
2. 同一份数据逻辑不应在多个页面重复维护。

## 3. 接入 Fake Store API：从 mock 到真实请求
### 做了什么
1. 建立了请求实例层（`http`）。
2. 建立商品 API 模块（`api/modules/product`）。
3. 引入数据适配：把第三方字段转换为项目统一字段。
4. `useProducts` 的数据来源切到真实 API。
5. 分类筛选从写死选项升级为接口返回。

### 实现思路
1. 先请求，再适配，再写入状态。
2. 页面永远使用统一后的字段（如 `name/price/category/image/stock`）。
3. 第三方字段差异在 API 层或适配层解决，不扩散到页面。

### 关键难点与处理
1. Fake Store 不提供 `stock`：当前用生成值方案先保证训练流程可跑。
2. 分类从本地枚举改为接口动态值，UI 绑定也随之调整。

## 4. 权限与登录链路升级（Day6-Day7）
### 做了什么
1. 用户登录状态保存在 Pinia，并落地本地存储。
2. 增加路由守卫：受保护页面要求登录。
3. 请求拦截器自动携带 token。
4. 响应拦截器处理 401，统一做登出和跳转。
5. 独立登录页 `LoginView`，实现：
   - 字段校验（用户名/密码）
   - 提交态（按钮禁用、防重复）
   - 就地错误提示（而非全靠 alert）
   - 登录成功后按 `redirect` 回跳

### 为什么这样做
1. 这是前端项目最常见的真实能力，不是语法练习。
2. 登录、守卫、HTTP 拦截器三者要形成闭环，而不是单点功能。
3. 表单体验好坏直接决定项目完成度和可用性。

### 关键认知
1. “路由守卫”负责页面入口权限。
2. “HTTP 拦截器”负责请求维度权限和过期处理。
3. “登录页表单状态”负责用户交互质量。

## 5. 列表交互升级（Day8）
### 做了什么
1. 在 `useProducts` 中加入排序状态（字段 + 顺序）。
2. 实现分页状态（页码、页大小、总页数、分页数据）。
3. 区分首次加载与手动刷新状态。
4. 页面增加空状态、错误重试、骨架加载。
5. 搜索/分类/排序变化时自动重置到第一页。

### 为什么这样做
1. 让列表页行为更接近生产系统，而不是静态展示。
2. 交互状态分离后，页面不会因为刷新操作频繁闪烁。
3. 排序/分页统一在 composable 管理，后续复用成本低。

### 关键认知
1. 过滤 -> 排序 -> 分页 是推荐处理顺序。
2. 总数统计应基于排序前/分页前的结果集，而非当前页。
3. 页面只渲染“最终结果”，不在模板层写复杂计算。

## 6. 今日架构层面的提升
今天最重要的不是某个按钮，而是你项目结构已经从“页面堆逻辑”转向“分层治理”：
1. `api` 层负责通信与数据进入。
2. `composables` 层负责领域状态与派生逻辑。
3. `stores` 层负责跨页面全局状态（用户、购物车）。
4. `views` 层负责交互呈现。

这意味着你已经在练“工程化前端”，不是单纯写页面。

## 7. 今日仍有的可接受遗留
1. `stock` 来源是模拟值，不是后端真实字段（当前阶段可接受）。
2. 详情页错误态与细粒度反馈还可继续完善。
3. 少量样式和事件绑定细节还有优化空间（不影响主线）。

3. 为结算失败加入可重试路径，避免用户丢失上下文。
