## 3. 路由能力：静态路由到动态路由
### 做了什么
1. 配置了静态页面路由（`/`、`/tasks`、`/profile`、`/practise`）。
2. 增加动态路由 `/practise/:id`。
3. 在列表页点击商品卡片时跳转详情页，形成“列表 -> 详情”链路。

### 实现思路
1. 路由层只负责“路径与页面组件映射”。
2. 业务层（列表页）通过事件拿到商品 `id`，调用 `router.push()` 跳转。
3. 详情页通过 `useRoute()` 读取 `params.id`，再根据 `id` 获取当前商品。

### 关键认知
1. `route.params.id` 是字符串，和 mock 数据中的数字 `id` 对比前需要类型转换。
2. 路由只是导航，不承载复杂状态，业务数据仍由页面/Store管理。

## 4. 组件通信：父子协作闭环
### 做了什么
`PractiseView`（父） + `PractiseSonView`（子）完成了完整通信：
1. 父通过 `props` 把 `product`、`categoryName` 传给子组件。
2. 子组件通过 `emit` 往父组件抛出 `add-to-cart`、`click-card` 事件。
3. 父组件接收事件并分别处理“加购”和“跳详情”。
4. 子组件用 `slot` 支持父组件注入“库存紧张”提示。

### 为什么这样做
1. 子组件保持“展示 + 触发事件”，不直接操控父层状态，职责清晰。
2. 父组件集中处理业务逻辑，后续重构到 Pinia 更容易。
3. `slot` 用于可变区域，提升组件复用性。

## 5. 列表页业务逻辑（PractiseView）
### 做了什么
1. 实现商品数据加载（mock + 异步模拟）。
2. 实现搜索、分类筛选。
3. 实现统计计算（总数、分类数、平均价、库存状态、高价商品判断）。
4. 实现购物车摘要展示（数量、金额、缺货检查、结算入口）。

### 方法层面练到了什么
1. `filter`：做搜索和分类筛选。
2. `reduce`：做总数量、总金额、平均价计算。
3. `some/every/find`：做状态判断和目标数据检索。
4. `computed`：把衍生数据和源数据分离，减少模板内重复逻辑。

## 6. 详情页业务逻辑（PractiseDetailsView）
### 做了什么
1. 读取路由参数 `id`。
2. 根据 `id` 找到对应商品并渲染详情。
3. 提供“加入购物车”按钮，调用 Pinia store 的 `addItem`。
4. 保留返回列表页入口，保证单路径流程清晰。

### 设计意义
1. 详情页从“静态展示”升级为“能操作状态”的真实业务页。
2. 页面动作直接连到 store，后续切换真实接口时改动更小。

## 7. 状态管理与持久化（Pinia + storage）
### 做了什么
1. 新建 `cart` store，维护购物车 `items`。
2. 实现 `addItem`、`clearCart`。
3. 实现 `totalCount`、`totalPrice` getter。
4. 初始化时从本地读取购物车，更新后写回本地。

### 核心思路
1. 把购物车当作“单一数据源”维护在 store 中。
2. 页面只触发 action 和读取 getter，不各自维护一份购物车。
3. 持久化只放在 store 内，避免页面层重复写 `localStorage`。

### 当前边界
1. 目前未实现 `removeItem/updateQuantity`，这是有意延后，不影响当前主线练习。
